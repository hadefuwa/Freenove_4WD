00.0_Servo_90 Center the Steering Servo ▶

This little program tells the car's steering servo to point straight ahead (90 degrees out of its 0–180 degree range) so you can check it's mounted correctly before doing anything fancier.

It teaches core C++/Arduino ideas: setup() runs once to get hardware ready while loop() repeats forever; functions can take parameters (like an angle in degrees) and be reused from other files via a header (.h) file; and how a servo motor is actually controlled using PWM — rapid on/off pulses whose width (measured in tiny fractions of a second) tells the servo which angle to move to.

01 — First Movesmotors, servos, sound, lights, and eyes
01.1 Car Move & Turn ▶

This program teaches the robot car to drive in every direction using its four special "mecanum" wheels, which have angled rollers that let the car go forward, spin in place, slide sideways, or glide diagonally — all without turning the wheels like a steering wheel.

It shows the setup()/loop() pattern, how a single function called Motor_M_Move controls all four wheels independently with speed numbers from -100 to 100, and how PWM is used to control motor speed. The real lesson is in the pattern of numbers passed to Motor_M_Move: matching signs make the car go straight, opposite left/right signs make it spin, and opposite front/back signs make it slide sideways or diagonally.

01.2 Servo Sweep ▶

This program controls a servo motor — a small motor that can turn to a specific angle and hold it, unlike a regular motor that just spins. The code first sets the servo to its middle position (90°), then sweeps it smoothly back and forth between 30° and 150°, forever, moving one degree at a time with tiny pauses in between.

It teaches the setup()/loop() structure, functions with parameters (like Servo_1_Angle(degrees) and Servo_Sweep(start, end)), and how PWM signals are used to tell real hardware what to do.

01.3 Buzzer ▶

This program makes the onboard buzzer beep out a pattern of sounds — 4 short beeps, repeated 3 times.

It teaches setup()/loop(), and introduces digital output — the idea that a computer pin can only be fully ON (HIGH) or fully OFF (LOW), and that flipping a pin HIGH and LOW very fast is exactly how the buzzer makes a musical pitch. It also shows how breaking a task into small, well-named functions (Buzzer_Setup, Buzzer_Alert, freq) makes code easier to read and reuse.

01.4 Battery Level ▶

This program measures how much charge is left in the battery and prints it to the computer screen every 0.3 seconds. It works by reading a voltage through a "voltage divider" circuit into the Pico's ADC (analog-to-digital converter), which turns a real-world voltage into a number between 0 and 1023, then uses a bit of math to convert that number back into real volts.

It shows off core C++ ideas: functions that take inputs and return a result (like Get_Battery_Voltage() returning a decimal volts value), and simple arithmetic (division, multiplication, averaging several readings) turning raw sensor data into something meaningful.

01.5 Matrix Eyes ▶

This sketch makes the car's two LED eye-matrices blink a cute animated face. Each 8×8 grid of tiny lights is drawn using bytes — one byte per row, where each of its 8 bits is either a 0 (LED off) or 1 (LED on), written compactly in hex like 0x18. The program stores several of these 8-row "frames" in arrays (x_array, y_array) and loops through them with a short delay between each, creating a simple flipbook-style animation.

It teaches arrays, binary/hex number representation, bit patterns, and using loops to repeat an action (showing each frame) over and over forever.

02 — Ultrasonic Sensingbat-style sonar, then obstacle avoidance
02.1 Ultrasonic Ranging ▶

This program makes the ultrasonic sensor "look around" by riding on a servo that swings it right, center, left, and back to center, over and over. Each time it stops at a new angle, the code sends out a tiny, silent sound pulse and times how long the echo takes to bounce back — just like a bat uses echolocation — then does some quick math (distance = time × speed of sound ÷ 2) to turn that time into a distance in centimetres.

Kids see how functions organize repeated actions (turning the servo, pinging the sensor), and how real-world sensors turn a physical measurement (time) into something useful (distance) through a formula.

02.2 Obstacle Avoidance Car ▶

This program makes the car act like a bat using sonar to feel its way around a room. Every loop, the car swivels its ultrasonic sensor left, straight ahead, and right to measure how far away things are, then uses if/else decisions: if the path is clear, keep driving forward; if something is too close on one side, curve away from it; if something blocks the way straight ahead, stop, back up, and spin toward whichever side has more open space.

It's a great first example of "sensor-driven decision making" — comparing numbers against thresholds (like OBSTACLE_DISTANCE) to let the robot choose its own actions instead of just following a fixed set of commands.

03 — Light Sensorsreading brightness, then steering toward it
03.1 Photosensitive (Light Reading) ▶

This program uses two photoresistors (light-sensing components) to check how bright it is on the left and right sides of the robot. Every half-second, it reads each sensor's light level as a number using analogRead(), then sends both numbers to the Serial Monitor so you can compare which side is brighter or darker.

It's a great first look at reading real-world analog sensors, storing values in variables, and using a loop() that repeats forever with a delay() in between.

03.2 Light-Tracking Car ▶

This program gives the car "phototropism" — like a plant bending toward sunlight, it steers itself toward whichever side has brighter light. Every fraction of a second it compares its two light sensors: if the left sensor sees more light, it speeds up the right wheels to turn left, and vice versa.

It teaches comparing two numbers to choose a direction, and using that comparison result (a "difference") to smoothly adjust motor speeds instead of just turning fully one way or the other — a nice first taste of proportional control instead of simple on-off logic.

04 — Line Trackingreading the floor sensors, then following a line
04.1 Tracking Sensor (Reading) ▶

This program checks the three infrared line sensors on the bottom of the car — Left, Middle, Right — which each report a simple 0 (white floor) or 1 (black line). It teaches digital input, and the neat trick of combining three separate 0/1 bits into a single number from 0–7 using bit-shifting and OR, so instead of juggling three values you get one number that tells you the exact pattern of what's under the car.

This groundwork — turning sensor readings into one combined number — is exactly what the next lesson uses to make the car follow a line automatically.

04.2 Line-Following Car ▶

This program is the "brain" of a line-following robot. Every loop it looks at the three floor sensors, squishes their readings into one number (0–7), and uses a switch/case to decide what that number means: drive straight, veer left, veer right, or stop at the finish line. It then spins the four wheels — sending opposite-signed speeds to the left and right sides to pivot like a tank — and shows a matching face or arrow on the LED display.

Kids learn switch/case decision-making, combining multiple sensor readings into one value ("sensor fusion"), and how paired positive/negative motor speeds create a turn instead of a straight drive.

05.2 IR Receiver Car (RC Driving) ▶

This program turns the car into a real RC car you drive with an infrared remote! Every button sends its own secret code, and handleControl() uses a big switch/case "lookup table" to match each code to a specific move — forward, back, turn, or (in a special mode) slide diagonally sideways, which only mecanum wheels can do.

It also teaches "non-blocking timing," where the car checks the clock (millis()) instead of freezing with delay(), so its animated LED face keeps blinking while it's still listening for your next button press. If no button code is found, the code falls through to a safe "default" case that simply stops the motors.

05.3 Multi-Functional Car (IR + Face + Lights) ▶

This is the "everything car" for this stage — it combines three things you built separately before: driving with the IR remote, showing feelings on the LED-matrix face, and lighting up the RGB strip. It teaches how real programs combine multiple systems at once using a single fast-repeating loop — checking the remote, updating the face, driving the wheels, and animating the lights all happen together, many times per second.

You'll see how a remote button press becomes a number your code reacts to, how "eyes" and lights are drawn frame-by-frame using the clock instead of freezing the program with delays, and how the car can automatically detect which accessories are plugged in.

06 — WiFi Controlcontrolling the car over the network from a phone app
06.1 WiFi TCP Server (Networking Basics) ▶

This program turns the Pico W into a tiny WiFi "phone" that only answers calls — it connects to your home WiFi and waits on a specific "door number" called a port (4002) for a companion phone/PC app to connect. Once connected, the Pico acts like a chat bridge: anything the app sends gets printed to the Serial Monitor, and anything you type gets sent back to the app.

It teaches WiFi networking basics (SSID/password, IP addresses, ports), how a TCP server waits for and accepts connections, and how programs check ("poll") repeatedly for new data using functions like client.connected() and client.available().

06.2 Multi-Functional Car (Full WiFi App Control) ▶

This is the "everything" program — it lets you control the whole car from a phone app over WiFi, all in one sketch! Your phone sends short text messages like N#45#80#0#0, and the Pico W reads that text, breaks it into pieces ("parsing"), and calls the right function to drive the mecanum wheels, change the LED face's expression, light up the RGB strip, or check the battery and sensors.

It teaches how a computer can run several "brains" at once without freezing — using millis() timers instead of delay() so the face animates, the lights glow, and new WiFi commands keep arriving all at the same time — plus it shows off some real trigonometry, letting a joystick's angle and distance become sideways, forwards, and spinning motion for the mecanum wheels.
